# 病原微生物防控系统 - 科室多人使用架构方案

> 针对科室多人协同使用场景的架构设计建议

---

## 🚨 单HTML架构的多人使用限制

| 限制项 | 问题描述 | 影响程度 |
|--------|----------|----------|
| **数据孤岛** | 每个同事的浏览器/手机数据独立，无法共享 | 🔴 严重 |
| **更新困难** | 修改后需通知所有人重新下载/更新 | 🔴 严重 |
| **数据冲突** | 多人录入同一监测点数据会重复 | 🟡 中等 |
| **无权限控制** | 无法区分录入员、审核员、管理员 | 🟡 中等 |
| **数据备份** | 同事离职或手机丢失，数据无法恢复 | 🔴 严重 |

---

## 🏗️ 推荐架构方案

### 方案一：内网服务器部署（推荐）

适合：科室有内网服务器或可以部署在办公电脑

```
┌─────────────────────────────────────────────────────────┐
│                      科室办公网络                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
│  │  廖所电脑  │  │ 同事A电脑 │  │ 同事B电脑 │             │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘             │
│       │              │              │                   │
│       └──────────────┼──────────────┘                   │
│                      │                                   │
│              ┌───────┴───────┐                          │
│              │  内网服务器    │                          │
│              │  (Node.js)    │                          │
│              │  + SQLite     │                          │
│              └───────────────┘                          │
└─────────────────────────────────────────────────────────┘
```

**技术栈**：
- **后端**：Node.js + Express
- **数据库**：SQLite（轻量，无需安装）
- **前端**：保持现有HTML，添加API调用
- **部署**：科室任意一台电脑（Windows/Linux均可）

**优点**：
- ✅ 数据集中存储，全员共享
- ✅ 自动更新（刷新网页即可）
- ✅ 权限可控（登录账号区分角色）
- ✅ 数据备份方便（定期复制sqlite文件）

---

### 方案二：文件服务器 + 数据同步（过渡方案）

适合：暂无服务器，但希望数据共享

**实现方式**：
1. 在共享文件夹（如NAS、科室公共盘）存放数据文件
2. 系统读取/写入JSON/CSV格式的数据文件
3. 前端仍然是一个HTML文件

```javascript
// 伪代码示例
async function loadMonitoringData() {
    // 从共享文件夹读取数据
    const response = await fetch('file:////192.168.1.100/shared/monitoring-data.json');
    return await response.json();
}
```

**限制**：
- ⚠️ 浏览器安全限制，需要特殊配置
- ⚠️ 文件锁问题（多人同时写入会冲突）
- ⚠️ 不适合高频操作

---

### 方案三：飞书/钉钉微应用（推荐无服务器场景）

适合：没有IT支持，希望快速上线

**实现方式**：
1. 利用飞书开放平台
2. 开发飞书小程序/机器人
3. 数据存储在飞书多维表格（Bitable）

**优点**：
- ✅ 无需部署服务器
- ✅ 飞书负责数据安全和备份
- ✅ 手机/电脑都能用
- ✅ 自带权限管理

**缺点**：
- ⚠️ 依赖飞书平台
- ⚠️ 定制化受限
- ⚠️ 数据在第三方平台

## 📊 各方案对比

| 维度 | 单HTML（现状） | 内网服务器 | 飞书微应用 |
|------|---------------|-----------|-----------|
| **部署难度** | ⭐ 极低 | ⭐⭐⭐ 中等 | ⭐⭐ 低 |
| **数据共享** | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| **权限管理** | ❌ 无 | ✅ 灵活 | ✅ 内置 |
| **数据安全** | ⭐⭐⭐ 高（本地） | ⭐⭐⭐ 高（内网） | ⭐⭐ 中（云端） |
| **扩展性** | ⭐ 差 | ⭐⭐⭐⭐ 好 | ⭐⭐⭐ 中等 |
| **维护成本** | ⭐ 极低 | ⭐⭐⭐ 中等 | ⭐⭐ 低 |
| **适合人数** | 1-2人 | 5-50人 | 无限制 |

## 🎯 我的建议

### 短期（1-2个月）：保持现状，做好准备

1. **代码模块化**：将数据和逻辑分离
   ```
   www/
   ├── index.html          # 主页面（瘦身）
   ├── js/
   │   ├── data/           # 数据文件
   │   │   ├── diseases.js
   │   │   └── disinfectants.js
   │   ├── modules/        # 功能模块
   │   │   ├── calculator.js
   │   │   ├── disease-plan.js
   │   │   └── monitoring.js  # 预留监测模块
   │   └── api.js          # API接口（预留）
   └── css/
       └── style.css       # 样式分离
   ```

2. **数据格式标准化**：
   - 监测数据使用统一JSON格式
   - 预留字段便于后续导入数据库

3. **试点使用**：
   - 先让1-2个同事试用
   - 收集需求和问题
   - 验证使用场景

### 中期（3-6个月）：内网服务器部署

如果试点成功，建议部署内网服务器：

**硬件需求**：
- 任意一台办公电脑（i5/8G内存即可）
- 固定IP或局域网域名
- 定期备份到移动硬盘

**开发计划**：
- 第1周：搭建后端框架
- 第2-3周：API接口开发
- 第4周：前端对接
- 第5周：测试上线

### 长期（6个月+）：功能扩展

- 病媒监测数据录入
- 自动报表生成
- 数据分析看板
- 与上级系统对接

## 💡 特别提醒

### 数据安全注意事项

1. **敏感数据**：
   - 病例信息不能存储（属于隐私）
   - 只存储消毒方案、监测数据等业务数据

2. **备份策略**：
   - 每天自动备份SQLite文件
   - 每周复制到外部存储
   - 每月验证备份可恢复

3. **权限设计建议**：
   ```
   管理员（廖所）
   ├── 系统配置
   ├── 数据审核
   └── 用户管理
   
   录入员（同事）
   ├── 数据录入
   ├── 数据查询
   └── 报表查看
   
   查看员（领导）
   └── 报表查看
   ```
